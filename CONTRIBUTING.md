# 贡献指南

感谢你愿意参与本项目！

## 开发流程建议

1. Fork 本仓库并创建分支：`feat/xxx`、`fix/xxx`。
2. 保持每次提交聚焦单一改动点。
3. 提交前请运行基础检查：
   - 后端：`mvn test`
   - 前端：`npm run build`（或至少 `npm run lint`）
4. 提交 PR 时请说明：
   - 变更内容
   - 变更原因
   - 兼容性影响
   - 验证方式

## 代码风格

- 尽量避免“超大函数”，优先可读性。
- 配置与业务逻辑分离，避免硬编码。
- 新增配置项请同步更新 `.env.example` 与 README。
- 注释优先写“为什么”，而不是“做了什么”。

## 与私有能力相关的约束

本项目为脱敏开源版：

- 不要在 PR 中尝试恢复内部专有逻辑。
- 对缺失能力请使用 Mock 或占位实现，并明确标注。
- 禁止提交任何密钥、令牌、私有端点、内部路径。
